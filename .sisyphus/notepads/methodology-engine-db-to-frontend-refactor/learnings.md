- Methodology facts dialog now uses max-h-[90vh] and overflow-y-auto on DialogContent to allow scrolling when content exceeds viewport, matching work unit behavior.
- Service architecture is layered around `MethodologyVersionBoundaryService` (L1 boundary alias of `services/methodology-version-service.ts`), which wraps core `version-service.ts` and adds lifecycle/agent/work-unit/workflow/artifact operations.
- `createAgent`, `updateAgent`, `createWorkUnitMetadata`, and `updateWorkUnitMetadata` all route through `updateDraftLifecycle` and perform full lifecycle snapshot validation via `validateLifecycleDefinition` before persistence; persistence uses `LifecycleRepository.saveLifecycleDefinition` (full replace of lifecycle tables).
- Granular lifecycle operations (e.g. state/transition/fact/workflow binding updates) use optional `MethodologyRepository` methods and usually call `coreService.validateDraftVersion` after write, rather than recomputing+replacing full lifecycle snapshot.
- `src/ports/*` tags (`VersionRepository`, `WorkUnitRepository`, `WorkflowRepository`, `MethodologyTx`) are exported but currently not wired into runtime layers; operational paths use `MethodologyRepository` and `LifecycleRepository` tags from `repository.ts` and `lifecycle-repository.ts`.
- Cardinality dropdown now marks the Contract tab dirty by calling `setIsContractTabDirty(true)` before handling form value changes, aligning with other fact fields and enabling discard warnings.
- Project pinning boundaries now accept `methodologyId` + `versionId` end-to-end; UI still fetches methodology details by key for read queries but all pin/repin mutations are ID-based to avoid secondary lookup indirection.
- `ProjectContextService` pin/repin validation now checks both methodology existence (`listDefinitions` by ID) and version existence (`findVersionById` + methodology ownership) before enforcing active-status compatibility.

- Workflow edge descriptions now come from the JSON column `descriptionJson` so repository writes and reads can pass the `{ markdown: string }` payload directly instead of converting to plain text.
- Tests and demo seeds now refer to `description_json` and store `{ markdown }` when exercising edge descriptions, so follow that pattern when adding edges to fixtures or SQL scaffolding.
