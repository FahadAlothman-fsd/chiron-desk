Must-fix issues:
1. Layer composition is incomplete. `packages/api/src/routers/project-runtime.ts` calls `InvokeWorkUnitExecutionService` (`startInvokeWorkUnitTarget`), but `packages/workflow-engine/src/layers/live.ts` does not provide `InvokeWorkUnitExecutionServiceLive` in `WorkflowEngineRuntimeStepServicesLive`. The runtime layer is therefore not composition-safe for the published API surface.
2. `packages/workflow-engine/src/services/invoke-workflow-execution-service.ts` is not transaction-safe/idempotent under failure or race. It creates a child workflow execution before marking the invoke target row started; if the second write fails or two callers race, orphan/duplicate child workflow executions can be created. This needs a single atomic repository operation or compensation path.
3. Both start services are missing a server-side active-step invariant. `invoke-workflow-execution-service.ts` and `invoke-work-unit-execution-service.ts` validate ownership, but neither rejects starts when the parent invoke step is no longer `active`. UI disabling is not enough for correctness.
4. `packages/workflow-engine/src/services/invoke-work-unit-execution-service.ts` depends on an ad-hoc optional `startInvokeWorkUnitTargetAtomically` method that is not part of `packages/workflow-engine/src/repositories/invoke-execution-repository.ts`. That weakens the Effect service boundary and leaves the live repository contract untyped/unverified.
5. Zero-target blocking logic is duplicated inconsistently. `packages/workflow-engine/src/services/invoke-target-resolution-service.ts` owns the canonical `No invoke targets resolved` state, but `packages/workflow-engine/src/services/invoke-completion-service.ts` re-derives eligibility without consulting it, so zero-target invoke steps can surface the wrong disabled reason. This violates the no-duplicated-business-state rule.
6. Test coverage is good at isolated service level (14/14 targeted tests pass), but it misses the critical integration/failure cases above: no live-layer composition test for invoke start services, no workflow-start race/mark-start failure test, and no zero-target completion-disabled test.
