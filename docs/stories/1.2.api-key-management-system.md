# Story 1.2: API Key Management System

## Status
Ready for Review

## Story
**As a** system administrator,
**I want** the Hono server to securely store and manage AI provider API keys and act as an API gateway that includes the appropriate API key in requests to the AI service,
**so that** users can safely use their own API keys while the AI service remains completely stateless and only receives the API key and provider information per request.

## Acceptance Criteria
1. Hono server database schema supports encrypted storage of API keys for multiple providers
2. Hono server exposes CRUD API endpoints for API key management (create, read, update, delete)
3. API keys are encrypted at rest using AES-256-GCM encryption in the Hono server
4. Hono server validates API keys before storing them (format verification, test call)
5. API key metadata includes provider, creation date, last used, and status
6. Support for API key rotation without service interruption
7. Audit logging for all API key operations in the Hono server
8. Soft delete for API keys to maintain audit trail
9. Hono server acts as API gateway, retrieving API keys from database and including them in requests to AI service
10. AI service remains stateless and receives API key and provider information in each request header

## Dev Notes

### Architecture Context
This story implements the API key management system following the established architectural pattern where the Hono server handles all data operations and the AI service remains stateless. [Source: architecture/server-service.md#Core Responsibilities]

### Database Schema
The existing database schema in the Hono server already includes the api_keys table: [Source: architecture/server-service.md#Database Schema]
```sql
CREATE TABLE api_keys (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    provider VARCHAR(50) NOT NULL, -- 'openrouter', 'anthropic', etc.
    key_hash VARCHAR(255) NOT NULL,
    key_preview VARCHAR(8), -- Last 4 characters for identification
    name VARCHAR(255),
    is_active BOOLEAN DEFAULT true,
    usage_limit INTEGER,
    usage_count INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT NOW(),
    expires_at TIMESTAMP
);
```
**Note**: The existing schema needs to be extended with `encrypted_key` column and audit logging table.

### Service Communication Pattern
The Hono server acts as an API gateway, retrieving API keys and forwarding requests to the AI service: [Source: epic-1-foundation-openrouter-integration.md#Service Communication Pattern]
```
Frontend → Hono Server → Database → AI Service → OpenRouter
     ↓           ↓            ↓         ↓           ↓
   UI Form    Get Encrypted   Include    Use Key    Make Request
            Key from DB    in Request  per Request
                          to AI Service
```

### Security Implementation
Use the existing security architecture for API key management: [Source: architecture/security.md#API Key Management]
- AES-256-GCM encryption with PBKDF2 key derivation
- Service-to-service authentication using HMAC-signed tokens
- API key validation via provider endpoints

### Technical Requirements

### Database Schema Extensions
```sql
-- Add encrypted_key column to existing api_keys table
ALTER TABLE api_keys ADD COLUMN encrypted_key TEXT NOT NULL;
ALTER TABLE api_keys ADD COLUMN last_used_at TIMESTAMP;
ALTER TABLE api_keys ADD COLUMN metadata JSONB;

-- Create usage logs table
CREATE TABLE api_key_usage_logs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    api_key_id UUID NOT NULL REFERENCES api_keys(id),
    used_at TIMESTAMP DEFAULT NOW(),
    endpoint VARCHAR(255),
    success BOOLEAN,
    error_message TEXT,
    tokens_used INTEGER,
    cost_usd DECIMAL(10,6),
    INDEX idx_api_key_usage (api_key_id, used_at)
);
```

### Encryption Requirements
- **Algorithm**: AES-256-GCM for authenticated encryption [Source: architecture/security.md#API Key Management]
- **Key Management**: Use environment variable for master key
- **Key Derivation**: PBKDF2 with 100,000 iterations and salt
- **Security**: Never log or expose unencrypted keys

### Provider Support
- **OpenRouter**: Format validation, test endpoint `/models`
- **OpenAI**: Format validation, test endpoint `/models`
- **Anthropic**: Format validation, test endpoint `/messages`
- **Future**: Extensible design for new providers

### API Gateway Pattern
- **Request Flow**: Client → Hono Server → AI Service → OpenRouter
- **API Key Injection**: Hono server retrieves encrypted key, decrypts, and includes in AI service request headers
- **Headers**: `X-API-Key: <decrypted-key>`, `X-API-Provider: <provider>`
- **Error Handling**: Graceful fallback when API keys are invalid or missing
- **Rate Limiting**: Applied per user/API key combination at the gateway level
