# Story CCF.2: Add Thin Electron Shell With Secure Runtime Boundaries

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a platform engineer,
I want a thin Electron shell with secure preload boundaries,
so that desktop hosting is available without rewriting renderer or backend architecture.

## Acceptance Criteria

1. `apps/desktop` is introduced as the desktop host, and when the Electron shell boots it hosts the existing renderer while `apps/web` remains the single renderer source.
2. Electron window configuration applies secure runtime defaults with `contextIsolation` enabled, `nodeIntegration` disabled, and renderer access available only through explicit preload APIs.
3. The Electron host is wired for both development and packaged startup paths so the shell can load the web renderer in dev and built renderer assets in packaged mode without rewriting renderer or backend architecture.
4. The modular architecture remains intact after the Electron shell is added: `contracts`, domain engines, transport packages, and `core` boundaries stay unchanged, and no domain package is moved into the desktop-specific layer.

## Dev Notes

- Treat `CCF.2` as the story that makes the Electron shell real and runnable, not the story that proves full runtime parity. `CCF.3` owns parity validation, reconnect hardening, startup edge cases, diagnostics follow-through, and mismatch cleanup after the shell exists.
- Preserve runtime ownership exactly as planned: `apps/web` remains the single renderer source and keeps existing client responsibilities, `apps/server` remains the shared backend/runtime host, and `apps/desktop` is a thin Electron container for window lifecycle, preload bridge, and local host orchestration only. Do not move product logic, domain logic, transport contracts, or `core` orchestration into the desktop layer. [Source: `_bmad-output/planning-artifacts/epics.md#CCF.2 Add Thin Electron Shell With Secure Runtime Boundaries`] [Source: `_bmad-output/planning-artifacts/architecture.md#High Level Architecture`] [Source: `docs/plans/2026-03-13-cc-foundation-electron-cutover-design.md#Architecture`]
- Follow the agreed desktop orchestration model in Electron main: `resolve -> probe -> attach if running -> start if absent -> wait for readiness -> fail clearly`. This should work like the OpenCode-style local-server pattern the user requested: Electron first checks for a compatible local backend, attaches when one is already available, and only spawns a managed local process when nothing acceptable is running. In dev this means it should coexist with `bun dev`; in packaged mode the same orchestration pattern applies, but Electron owns launching or attaching the local backend process without assuming Bun-based developer workflows.
- Use OpenCode as the implementation precedent for orchestration shape only, not as a literal desktop architecture template. Its VS Code integration starts or attaches to a local process and polls HTTP readiness before continuing, which maps well to Chiron's `apps/server` lifecycle needs; its terminal/TUI session model does not map to Chiron because this shell must load `apps/web` and coordinate `apps/server` from Electron main. [Source: `https://github.com/anomalyco/opencode`] [Source: `https://github.com/anomalyco/opencode/blob/dev/sdks/vscode/src/extension.ts`] [Source: `https://opencode.ai/docs/cli/`]
- Prefer an HTTP health probe as the readiness signal before normal renderer load. A mere port-open check is too weak for this cutover story because the desktop shell needs a deterministic signal that the backend is actually usable before `apps/web` continues with normal runtime behavior.
- If startup fails or readiness times out, require a clear blocking startup error with recovery guidance instead of silently retrying or letting the renderer fail later through generic network errors. Capture structured diagnostics for bootstrap, preload boundary status, dependency attach/start decisions, readiness failures, and desktop runtime model evidence so later stories can harden and validate from concrete signals rather than guesswork.
- Keep the first preload bridge real but narrow. It is not the backend and not a generic IPC tunnel; it is the safe boundary Electron exposes to the renderer through `contextBridge`. For `CCF.2`, the bridge should expose runtime info plus a tiny set of explicit host controls only, such as runtime status and a narrow recovery action like restarting required local services. Do not expose raw Node APIs, broad Electron primitives, generic command channels, or direct process control to `apps/web`. [Source: `docs/plans/2026-03-13-cc-foundation-electron-cutover-design.md#Data Flow And Runtime Boundaries`] [Source: `_bmad-output/project-context.md#Implementation Guardrails`]
- Apply secure Electron defaults at window creation: keep `contextIsolation` enabled, `nodeIntegration` disabled, and renderer access limited to explicit preload APIs. Current Electron security guidance also supports using preload plus `contextBridge` as the safe renderer boundary; avoid broad renderer privilege even if the shell only targets local content. [Source: `_bmad-output/planning-artifacts/epics.md#CCF.2 Add Thin Electron Shell With Secure Runtime Boundaries`] [Source: Context7 `/electron/electron` security docs]
- Follow the test-first migration pattern established in `CCF.1`: create narrow package-local tests that fail before implementation and pass afterward. For `CCF.2`, keep tests focused on the desktop host contract: secure `BrowserWindow` defaults, dev URL resolution, packaged renderer path resolution, backend URL/health-probe orchestration, and blocking failure behavior. Do not widen this story into full parity smoke, Playwright desktop interaction, or broader framework-gate coverage yet; those belong to later CC-Foundation stories.
